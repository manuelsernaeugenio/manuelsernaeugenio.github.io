Tecnologías web

Introducción

En el desarrollo web intervienen diferentes tecnología que han de utilizarse de forma combinada para un buen funcionamiento de las aplicaciones. Se pueden clasificar dichas tecnologias en:

  • Tecnologías de cliente. Su función principal es la de realizar forma la petición HTTP a partir de los parámetros necesarios, enviarla, recibir los datos, procesarlos y presentarlos. La mayor parte de estas tecnologías se aplican en los navegadores.

  • Tecnologías de servidor. En este caso su función principal es recibir las peticiones, procesar dicha petición y de volver la respuesta al cliente. Dentro de estas tecnologías se tienene:

    • Servidores web. Especializados en el protocolo HTTP.

    • Servidores de apliciones. Más especializados que los web, pueden trabajar con HTTP y otros protocolas, además de facilitar infraestructura para funcionalidades comunes.

    • Lenguajes de servidores. Su función es a partir de la petición tratada por el servidor, realizar las acciones necesarias y fabricar la respuesta.

    • Otros servidores. Conjunto de protocolos y servidores auxiliares necesrios para el funcionamiento de las aplicaciones web.

  • Tecnologías de despliegue. El desarrollo de aplicaciones web y su puesta en producción es una tarea compleja. Se han desarrollado diferentes aplicaciones en función de los servidores y lenguajes que automatizan en mayor o menor grado la puesta en marcha de una aplicación web.

Tecnologías del cliente

Los clientes web clásicos han sido los navegadores, aunque desde hace unos años pueden ser otros servidores, aplicaciones nativas de los diferentes sistemas operativos, «scripts», aunque los navegadores siguen siendo los más utilizados. Las tecnologías utilizadas en estos clientes son:

Lenguaje de marcas que el navegador recibe, va leyendo, crea un árbol con nodos y los representa en la pantalla. Los nodos HTML pueden ser:

  • Document. Nodo ráiz del árbol.

  • Element. Nodos a partir de las etiquetas HTML como <div> o <a>.

  • Text. Nodos de texto, hijos normalmente de nodos de tipo Element.

  • Attributos. Nodos hijos de Element que almacen personalizaciones de Element como src o style.

La versión actual es HTML5, ha introducido algunos cambios significativos con versiones anteriores:

  • Web semántica.

    Anteriormente la estrucutra de una aplicación se realizaba utilizando <div>, esto aporta poca información semática (parte de la lingüística que estudia el significado de las expresiones lingüísticas). En esta nueva versión se añaden etiquetas, con el mismo comportamiento de <div> pero que añade información extra sobre la página. Algunos ejemplos:

    • <article>

    • <section>

    • <video>

    • <audio>

    • <progress>

    La web semática permite que las web sean analizadas no solo por humanos, sino por programas. Además facilita la accesibilidad.

A partir de estas necesidades, en HTML5 se han definido una serie de API’S para facilitar el desarrollo de aplicaciones más potentes que las clásicas aplicaciones web, de forma que se pueda acceder a información de la máquina/usuario, mejorar el rendimiento y almacenar gran cantidad de información en el cliente. Algunas de las API`s definidas:

    • Drag and Drop. Para dispositivos con pantallas táctiles, funciona con eventos.

    • Trabajo offline. Por ejemplo Google Docs, se basa en el uso de cachés e indicaciones de páginas web necesarias para funcionar offline.

    • Geolocalización. A través del navegador solicita las coordenadas actuales. El navegador suele pedir permiso previamente.

    • Almacenamiento. Hasta HTML5, la única forma de almacenar información persistente en el cliente eran las «Cookies». Con esta nueva versión se definen un almacen persistente de tipo clave/valor

Introducido por el antiguo navegador Netscape (precursor de Firefox), su función inicial era la de modificar el árbol DOM HTML e interactuar con ciertas funciones del navegador y validar formularios. Inicialmente llamado LiveScript, pero que paso a llamarse Javascript al llegar a un acuerdo con Sun.

En 1997 Netscape decide estandarizar el lenguaje a través de ECMA, naciendo el lenguaje ECMASript, aunque coloquialmente se sigue llamando Javascript.

Inicialmente sigue el paradigam estructurado, no orientado a objetos, sino a prototipos para la herencia, siendo el desarrollo y mantenimiento de los programas complejo , por ejemplo los métodos son funciones, y el constructor es una función. Un ejemplo de clase persona:

1 function Persona(nombre, apellido, edad, genero, intereses) {
2    // definiendo de propiedades y métodos
3    this.first = first;
4    this.last = last;
5    //...
6 }
7 var person1 = new Persona('Bob', 'Smith', 32, 'hombre', ['music', 'skiing']);

Desde sus inicios hasta la actualidad ha evolucionado hasta convertirse en uno de los lenguajes más usados Índice TIOBE, pasando por diferentes fases, algunas de ellas solapadas en el tiempo:

  • Guerra de navegadores. En los primeros años, ni HTML, ni Javascript se encuentran estandarizados. Los dos grandes navegadores (Firefox e IE), en especial este último no cumple con los mínimos estándares, época en la que desarrollar web era un «inferno».

  • Aparición primeras librerías. Se abstrae del navegador y aparecen librerías que intentan traer el mundo de los componentes visuales de las aplicaciones de escritorio a la web (Dojo o ExtJS), aunque la librería JQuery, orientada al manejo del DOM pasa a ser un estándar en el desarrollo Web.

  • Uso de peticiones asíncronas. Casi al mismo tiempo de la aparición de las primeras librerías, se comieza a utilizar una cualidad de los navegadores, las peticiones asíncronas con la interfaz XMLHttpRequest desarrollada por Microsoft, popularizándose el uso de servicios web en el cliente.

  • Frameworks, herramientas y aplicaciones SPA. Aparición de frameworks para la creación de aplicaciones completas en el navegador en una única página (SPA), usando servicios web. Los más conocidos (Angular, React y Vue), integrando HTML, Javascript y CSS. Además de herramientas para gestion de paquetes como NPM o calidad de código como ESLint entre muchas otras.

Tecnologías en el servidor

Elemento básico en las tecnologías web, en su versión más básica es un programa que se encuentra escuchando peticiones del protocolo HTTP en un puerto TCP, normalmente el 80. Inicialmente devolvía el recurso indicado en la petición, usualmente HTML, desarrollándose posteriormente la capacidad de pasar la petición a guiones o script externos (CGI) y cuya salida estándar es redirigida al cliente.

Un ejemplo sencillo de CGI con Bash:

#!/bin/bash

echo "Content-type: text/html"
echo ""
echo '<html>'
echo '<head>'
echo '<meta http-equiv="Content-Type" content="text/html; charset=UTF-8">'
echo '<title>Hello World</title>'
echo '</head>'
echo '<body>'
echo 'Hello World'
echo '</body>'
echo '</html>'
exit 0

Existen multitud de servidores web, incluso en los nuevos «frameworks» de desarrollo en diferentes lenguajes se incluye un servidor de pruebas. Estos servidores han ampliado su funcionalidad con el paso del tiempo sitios virtuales, gestión de usuarios, localización, SSL, caché, LDAP…

Los más usados:

  • Apache. Servidor más usado en la actualidad, inicialmente para Unix, existen versiones para la mayor parte de los sistemas operativos. Escrito en C, su principal característica es la velocidad gracias a poder añadir y quitar funcionalidades usando módulos. Actualmente se encuentra en la versión 2.4.53.

  • Nginx. Segundo servidor más usado, se combina bien con Apache ya que puede funcionar como proxy inverso. Usa menos memoria que Apache y también puede cargar módulos de forma dinámica. Algunas de sus características más destacadas:

    • Proxy inverso.

    • Web Caché.

    • Balanceo de carga.

    • Soporte HHTP2 y experimentar 3.

    • Proxy de correo POP3/IMAP.

  • Internet Information Service. Servidor de Microsoft para las versiones server de su sistema operativo. Muy integrado con sus lenguajes de programación C# o VB, para la web con ASP.Net, aunque puede soportar otros lenguajes como Python o PHP.

La línea que separa los servidores web de los servidores de aplicaciones son difusos. Se define un servidor de aplicaciones como un sistema distribuido que proporciona un entorno de ejecución para un programa de aplicación, pero esta definición también se puede dar actualmente en lo servidores web. Los servidores de aplicaciones se encuentran asociados a la tecnologia Java, en concreto Java EE ahora conocido como Jakarta EE, estos servidores han de cumplir con un estándar para poder ser certificados, proporcionando una serie de servicios adicionales que un servidor web no proporciona, como:

  • Seguridad

  • Registro

  • Persistencia

  • Gestión de recursos

  • Herramientas administración y/o configuración

  • Colas de mensaje

  • XML

No solo Java posee servidores de aplicaciones, si bien el término en el entorno profesional se asocia a Java, Microsoft con su plataforma .NET e IIS también implementa la filosifía de servidor de aplicaciones.

En Jakarta Enterprise Edition se tiene diferentes tipos de archivos que encapsulas los elementos software, junto con ficheros especiales que aportan información extra para configurar la aplicación y personalizar los recursos. Los tipos de archivos son:

  • JAR. Contienen la lógica de negocio y el modelo con los EJB (Enterpriese Java Beans), que son un tipo especial de clases similares a los JavaBeans, pero que añaden funcionalidades extra como comunicación remota, transacciones, concurrencia o seguridad.

  • WAR. En este caso para servidores certificados con el perfil. Tiene un archivo web.xml con la configuración en la carpeta WEB-IF.

  • EAR. Ficheros comprimidos con toda la aplicación, para servidores de aplicaciones cerficados con el perfil Web. Puede contener ficheros JAR y WAR. En la carepta META-INF se tienen diferentes ficheros de configuración como application.xml (obligatorio), icon o security-role.

Su función es a partir de la peticón HTTP entregada por el servidor junto con cualquier otra información disponible y necesaria, procesar dicha peticiónaciendo,

realizar las acciones necesarias (base de datos, servicios web…), fabricar la respuesta y devolverla al servidor para que la envie al cliente.

A lo largo del tiempo se ha producido también una evolución en los lenguajes de servidor:

Inicios. En las primeras páginas web (no podía considerarse como aplicación) los contendidos eran estáticos, es decir, un simple fichero HTML que el servidor leia y enviaba al cliente. Para modificar el contenido de la web era necesario editar a mano el contenido del HTML.

Primeros lenguajes en el servidor. La siguiente evolución fue llamar a guiones con lenguajes del propio sistema operativo conocidos como CGI, por ejemplo usando Bash. Esta solución tenía una serie de problemas muy importantes, como el manejo de QueryString, la comunicación con base de datos o la escritura usando la salida estándar de código HTML.

Lenguajes especializados. Aparecen los primeros lenguajes o adaptaciones de lenguajes ya usados para la generación de páginas web. Sucede a principios de siglo y solucionan en parte las carencias de los CGI. Aparece el concepto de sesión, aplicación, obtención de forma automática de los parámetros de entrada o integración de HTML y lenguaje de servidor ( Consiste en definir una página web en HTML e insertar en los lugares necesarios código especiales que son ejecutados.). Los lenguaes más destacados de esa época fueron (algunos siguien siéndolo).

  • PHP. El servidor llama a un intérprete, que va analizando el fichero y cuando encuentra la apertura de código <? ejecuta dicho código hasta encontrar el cierre ?>. Inicialmente era un lenguaje estructurado.

  • ASP. Tecnología de Microsoft para funcionar sobre IIS. El lenguaje era VisualBasic. Sustituido por ASP.NET.

  • JSP. Previamente a los JSP aparecieron los servlets, estos son clases pensadas para responder a solicitudes Http de forma similar a los CGI. La evolución fueron los JSP, que de igual forma que PHP o ASP permiten escribir HTML insertando código ejecutable dentro de la etiqueta: <% código Java … <%>. La primera vez que se ejecuta el JSP se «compila» a un servlet.

   /**

Evolucion. Para pequeñas aplicaciones mezclar HTML, CSS y lenguajes de servidor puede ser manejable aunque poco mantenible, en aplicaciones grandes es prácticamente imposible el desarrollo y mantenimiento, apareciendo nuevos lenguajes y entornos de trabajo o «frameworks» que dotan a las aplicaciones web de estructura y llevan la ingeniería del softaware a la web. Destaca también la aparición del modelo o patrón Modelo-Vista-Controlador o MVC y la división entre Front-End y BackEnd.

Además de un servidor, ya sea de aplicaciones o web y un cliente son necesarios otros elementos para su uso, en especial servidores que ofrecen servicios relacionadas con la web. Los principales servidores que completan la arquitecturas web son:

  • Servidor de ficheros. FTP. Servidor que permite la transferencia de ficheros entre máquinas, por ejemplo subir o actualiza el sitio web en el servidor web.

  • Servidor de directorio. Existen diferentes servidores de directorio, siendo el más conocido OpenLDAP que implemente el protolocolo LDAP. Su función es la de almacenar y recuperar información jerárquica, con muchas más lecturas que escrituras. Tiene múltiples usos (por ejemplo Active Directory), pero en la web se usa como elemento centralizado para la validación de usuarios en varias aplicaciones y/o sistemas.

  • Servidor de nombres. DNS.Los equipos se identifican con su dirección IP, pero a los humanos les cuesta recordar números y menos palabras, el servidor de nombres o DNS es un sistema jerárquico distribuido que permite dada una ip obtener el nombre y a la inversa, dada un nombre obtener la ip. Recordar que en el paquete IP la dirección es una IP no un nombre.

  • Servidor de correo. No relacionado directamente con la web, pero si ampliamente utilizado, puede implementar los protocolos SMTP, POP e IMAP.

  • Servidor de hora. NTP. Utilizado para la sincronización de la hora de otros equipos, en especial servidores, por ejemplo en un «cloud»

Existen muchos otros servidores como SSH, Telnet , de base de datos, irc, propios (puerto >1023)…

Tecnologías de despliegue

El despliegue de aplicaciones tanto nativas como Web a pasado se ser simplemente una compilación a convertirse en un factor imporante en la indústria informática. Algunas de las tareas típicas en el desarrollo de una aplicación web son:

  • Desarrollar el front-end.

  • Desarrolalr el back-end.

  • Compilacion, transpilación, empaquetado…

  • Pasar test independientes.

  • Añadir los cambios a los repositorios

  • Integrar.

  • Pasar test conjuntos.

  • Si los test han pasado, desplegar en servidores de preproducción o pruebas más detallados.

  • Pasar a producción.

Estas tareas llevan tiempo y pueden dar problemas en caso de no hacerse correctamente, existiendo diferentes herramientas como:

  • Jenkins: Servidor de automatización, muy utilizado permite definir flujos de trabajo con gestor de versiones, herramientas de compilación como Mave o Ant, línea de comandos…

  • Deployer: Para PHP, permite especificar las versiones que se suben al servidor e intercambiarlas.

  • Gulp: Para Javascript, aunque se puede usar en diferentes lenguajes, herramienta para la automatización de tareas.

  • Octopus Deploy: Similar a las anteriores, de pago, encaminado a DevOps e integración continua.

  • Bamboo: Herramienta de automatización para una integración continua.

  • GitHub Actions: Automatización de todos sus flujos de trabajo de software. Definen diferentes conceptos como flujo de trabajo, eventos o trabajos.
