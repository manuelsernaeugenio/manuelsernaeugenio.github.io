Modelos de arquitecturas web

Introducción

Las aplicaciones web poseen un modelo cliente/servidor, pero dentro de cada uno de estos se establecen diferentes arquitecturas. Este apartado se centra en las arquitecturas del BackEnd, aunque en el FrontEnd también se tiene arquitecturas, muchas de ellas iguales o similares a las del lado del servidor. En los siguientes puntos se describen algunas de Las principales arquitecturas usadas en Web en el servidor, aunque su uso se puede extender (y de hecho se extiende) a aplicaciones de escritorio y/o nativas, teniendo en cuenta que el protocolo de comunicación es HTTP, ya sea una aplicación nativa, el navegador u otro sistema.

Arquitectura monolítica

Fue la primera arquitectura, toda la funcionalidad se encuentra en el mismo fichero, mezclando la lógica, el acceso a datos, el código HTML o el estilo. Difícil de desarrollar, de depurar y mantener, genera código «espagueti».

Arquitectura Modelo-Vista-Controlador o MVC

Esta arquitectura ya se ha tratado en diferentes puntos del tema. Existe en la literatura diferentes definiciones de cada uno de los elementos de la arquitectura y con diferentes implementaciones. Divide la arquitectura monolítica en 3 elementos:

  • Modelo. Representa la información con la que se opera el sistema, en Java los típicos JavaBeans con getter y setter, junto con los mecanismos de persistencia y la lógica de negocio, dependiendo de la implementación es el resposible de pasar la información a la vista con los cambios.

  • Vista. Encargado de realizar las peticiones al controlador y esperar la respuesta de los cambios en el modelo para mostrarlo.

  • Controlador. Su función es la de a partir de la solicitud de la vista, redirigir la petición de la vista hacia el modelo correcto (posee en la web un enrutador). Dependiendo de la implementaciónaciendo crear una nueva vista (HTML) a partir de modelo.

Las principales variaciones entre las diferentes implementaciónes está en quien notifica los cambios en el modelo a la vista, si el propio modelo o el controlador, además existe un debate si el modelo se ha de implementa con objetos o interactuar directamente con la base de datos sin pasar por objetos. Variaciones de la arquitectura MVC

A partir de el modelo MVC se han desarrollado variaciones del mismo para adaptarlos a las necesidades de cada tipo de aplicación. Algunos de estas adaptaciones son:

  • HMVC. Consiste en poder tener una jerarquía de pequeñas arquitecturas MVC, de forma que se va «llamando» desde un MVC a sus hijos.

  • MVA. En este caso el modelo y la vista no se comunican en ningun caso, el adaptador (A) se encarga de transformar los datos del modelo a las necesidades de la vista.

  • MVP. Se descarga de responsavilidad a la vista, y es el presentador (P) en encargado de actuar sobre el modelo y formatear los datos para la vista.

  • MVVM. Model-View-ViewModel.La variante que más uso tiene, ya que Microsoft lo ha implenentado en aplicaciones de escritorio. Su principal diferencia es poder realizar pruebas unitarias para el modelo vista y poder evolucionar de forma independiente a la vista, de igual forma la vista puede evolucionar de forma independiente

Domain Driven Desing (DDD) y arquitectura hexagonal

En los últimos tiempos se han desarrollado nuevas arquitecturas, aunque DDD no es exactamente una arquitectura, sino más bien un enfoque en el desarrollo del software, que se centra en el problema a resolver (clases, entidades, algoritmos…), dejando en un segundo plano otros aspectos como las conexiones con las bases de datos, la gestión de los logs o configuración de las herramientas. Esta filosifia puede conseguir un código más limpio, la realización de pruebas de forma independiente de los diferentes elementos, además el acoplamiento entre los diferentes elementos de la aplicación es mínimo.

Para conseguir aislar el modelo del dominio de los elementos externos se utilzan diferentes alternativas, siendo la más conocida la arquitectura hexagonal. Esta arquitectura define diferentes elementos como:

  • Infraestructura. Elementos externos con los que se comunica la aplicación.

  • Puertos. Interfaces públicas para interactuar con la aplicación, puede ser un cliente o una base de datos, entre otras.

  • Adaptadores. Especialización de un puerto, realiza una transformación

Establece 2 capas independientes:

  • Domain. Contiene el dominio, las entidades y la lógica de negocio, nada más.

  • Entidades exterinas/infraestructura. Se comunican con el dominio a través de los puertos y adaptadores, siendo tanto la presentación como la base de datos o cualquier otro elemento externo.

Profundizar

En el artículo se profundiza en los fundamentos de la arquitectura hexagonal (originariamente pPorts and Adapters) .

También existe un curso en OpenWebinars sobre arquitectura hexagonal.

Por último indicar que la arquitectura hexagonal es un tipo de arquitectura que sigue las llamadas arquitecturas limpias o «clean architecture» que buscan la independencia de las capas y el menor acoplamiento posible, estableciendo una serie de limitaciones en cuanto a las capas de la aplicación y como regla general se estable la dependencia entre ellas que va de fuera hacia dentro, de forma que la capa más interna no dependa (acoplamiento) de ninguna otra capa, en la arquitectura hexagonal la capa de dominio.
